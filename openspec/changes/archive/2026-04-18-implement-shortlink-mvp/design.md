## Context

ShortLink 是一个从零开始构建的 URL 缩短服务。当前项目只有需求文档和技术设计文档，无可执行代码。

**技术栈**：
- Go 1.21+
- Gin（Web 框架）
- GORM（ORM）
- MySQL 8.0+（数据库）
- Redis 7.0+（缓存）
- Zap（日志）
- Viper（配置管理）

**架构模式**：分层架构（Handler → Service → Repository）

**约束**：
- 重定向响应时间 P99 < 50ms
- 短码长度固定 6 位
- 支持高并发（重定向 QPS ≥ 10,000）

## Goals / Non-Goals

**Goals:**
- 实现 URL 缩短核心功能（POST /api/v1/links）
- 实现短链重定向功能（GET /{code}）
- 达到性能指标（重定向 P99 < 50ms）
- 保证短码唯一性（Snowflake 算法）
- 实现缓存策略（命中率 > 95%）
- 提供完整的测试覆盖

**Non-Goals:**
- 用户认证系统（MVP 阶段暂不实现）
- 访问统计功能（后续迭代）
- 自定义短码（后续迭代）
- 消息队列集成（后续迭代）
- K8s 部署配置（仅提供 Docker）

## Decisions

### 1. 短码生成算法：Snowflake + 62 进制

**决策**：使用 Snowflake 算法生成唯一 ID，然后转换为 62 进制短码。

**理由**：
- 支持分布式部署（workerID 区分节点）
- 短码有序（便于排查问题）
- 碰撞概率极低（4096 QPS/毫秒）
- 无需数据库自增 ID

**备选方案**：
- 自增 ID 转 62 进制：需要分布式 ID 生成器，不适合多节点
- Hash(MD5)：碰撞处理复杂，长度固定
- 随机字符串：碰撞概率需计算，不可预测但无序

### 2. 缓存策略：Cache-Aside + Write-through

**决策**：
- 读操作：先查缓存，未命中查数据库并写入缓存
- 写操作：写数据库同时写缓存（Write-through）
- 空值缓存：短码不存在时缓存 "NULL"（TTL 5 分钟）
- 随机 TTL：24 小时 ± 10% 防止缓存雪崩

**理由**：
- 实现简单，性能优秀
- 数据一致性好
- 防止缓存穿透和雪崩

**备选方案**：
- Cache-Aside only：写操作不同步缓存，可能导致短暂不一致
- Read-through：需要实现复杂回调逻辑

### 3. 数据库 ORM：GORM

**决策**：使用 GORM 作为 ORM 框架。

**理由**：
- Go 生态最流行的 ORM
- 自动迁移（AutoMigrate）简化数据库初始化
- 支持关联查询、钩子函数
- 社区活跃，文档完善

**备选方案**：
- sqlc：类型安全，但需要手写 SQL
- 原生 database/sql：灵活但开发效率低

### 4. 项目结构：标准 Go 布局

**决策**：采用 `cmd/` + `internal/` + `pkg/` 标准结构。

**理由**：
- Go 社区标准实践
- `internal/` 限制包可见性
- `pkg/` 存放可复用组件
- 清晰的职责划分

### 5. 限流策略：Token Bucket（单机）

**决策**：使用内存 Token Bucket 算法实现限流。

**理由**：
- 实现简单，无外部依赖
- 适合 MVP 阶段
- 支持平滑限流

**备选方案**：
- Redis 限流：支持分布式，但增加复杂度
- Sliding Window：更精确但实现复杂

## Risks / Trade-offs

### 1. 短码碰撞风险
**风险**：Snowflake 算法理论上不会碰撞，但实现 bug 可能导致重复。
**缓解**：数据库唯一索引兜底，冲突时重试生成。

### 2. Redis 故障降级
**风险**：Redis 故障导致缓存失效，数据库压力增加。
**缓解**：降级到数据库查询，Redis 持久化（RDB + AOF），监控告警。

### 3. 数据库连接池耗尽
**风险**：高并发时连接池耗尽，请求阻塞。
**缓解**：合理配置连接池（MaxOpenConns=100），慢查询优化，监控连接数。

### 4. 缓存穿透
**风险**：大量查询不存在的短码，穿透到数据库。
**缓解**：空值缓存（TTL 5 分钟），布隆过滤器（后续优化）。

### 5. 单机限流局限
**风险**：多节点部署时，单机限流无法全局控制。
**缓解**：MVP 阶段可接受，后续迁移到 Redis 限流。

## Migration Plan

### 部署步骤
1. 初始化 MySQL 数据库（执行 GORM AutoMigrate）
2. 启动 Redis 服务
3. 部署 ShortLink 服务（Docker 或直接运行）
4. 健康检查：`GET /health`
5. 功能验证：创建短链 → 重定向测试

### 回滚策略
- 保留上一版本 Docker 镜像
- 数据库迁移向下兼容（仅新增表/字段）
- 快速回滚：切换 Docker 镜像标签
